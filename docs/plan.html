1Windows Server AD 버전별 특징 비교

Windows Server 2016 → 2019 → 2022 순서로 AD 기능이 어떻게 발전했는지 정리합니다. 도메인/포레스트 기능 수준(Functional Level)은 2016 이후 새로운 레벨이 추가되지 않았습니다.

ℹ️
도메인 기능 수준 (DFL / FFL) Windows Server 2019, 2022는 새로운 기능 수준을 추가하지 않았습니다. 최신 기능 수준은 여전히 Windows Server 2016입니다. 2019/2022의 AD 개선은 기능 수준과 무관한 개별 기능 추가 형태로 제공됩니다.

2Samba 4 vs Windows AD 기능 대조

Samba 4가 Windows Server AD의 핵심 프로토콜을 오픈소스로 재구현한 구현체입니다. 최대 Windows Server 2016 기능 수준까지 지원합니다.

5구축 단계별 계획

P1
Phase 1 Samba 4 AD DC 설치 및 도메인 프로비저닝
bash
# 패키지 설치
apt update && apt install -y samba krb5-user winbind libnss-winbind libpam-winbind

# 기존 smb.conf 제거
mv /etc/samba/smb.conf /etc/samba/smb.conf.bak

# 도메인 프로비저닝
samba-tool domain provision \
  --use-rfc2307 \
  --domain=TEST \
  --realm=TEST.LOCAL \
  --server-role=dc \
  --dns-backend=SAMBA_INTERNAL \
  --adminpass='Admin@Test1234!'

# Kerberos 설정 복사
cp /var/lib/samba/private/krb5.conf /etc/krb5.conf

# Samba 시작
systemctl unmask samba-ad-dc
systemctl enable --now samba-ad-dc
  • 프로비저닝 완료 후 samba-tool domain info 127.0.0.1 으로 도메인 정보 확인
  • DNS 동작 확인: host -t SRV _ldap._tcp.test.local 127.0.0.1

Phase 2 Windows PC 도메인 조인 테스트

    Windows PC DNS 설정 제어판 → 네트워크 → IPv4 속성 → DNS 서버를 VM IP(192.168.1.100)로 변경

    파일 공유 접근 테스트 탐색기에서 \\dc01.test.local\sysvol 접근 — Kerberos SSO 동작 확인

#!/usr/bin/env python3
"""
대량 계정 생성 스크립트 — samba-tool 래핑
사용법: python3 bulk_users.py --count 10000 --prefix testuser
"""
import subprocess
import argparse

def create_user(username, password="Test@1234!"):
    result = subprocess.run(
        ["samba-tool", "user", "create", username, password,
         "--given-name", username, "--surname", "Test"],
        capture_output=True, text=True
    )
    return result.returncode == 0

def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--count", type=int, default=1000)
    parser.add_argument("--prefix", default="testuser")
    args = parser.parse_args()

    success = 0
    for i in range(args.count):
        username = f"{args.prefix}{i:06d}"
        if create_user(username):
            success += 1
        if i % 100 == 0:
            print(f"진행: {i}/{args.count} (성공: {success})")

    print(f"완료 — 총 {success}/{args.count} 계정 생성")

if __name__ == "__main__":
    main()
⚠️
10만 계정 생성 시 samba-tool subprocess 방식은 느립니다 (계정당 ~50ms → 총 ~1.5시간). 빠른 생성이 필요하면 python-ldap 직접 LDAP Add 방식으로 전환 권장 (10배 이상 빠름).

# 시나리오 검증 방법 기대 결과
1 Windows PC 도메인 조인 + AD 계정 로그인 Windows 로그인 화면에서 AD 계정 입력 정상 로그인
2 Windows PC → 파일 공유 SMB Kerberos SSO 탐색기에서 공유 폴더 접근 (비밀번호 입력 없이) 자동 인증 후 폴더 열림
3 그룹 기반 파일 공유 접근 제어 그룹 미소속 계정으로 공유 접근 시도 접근 거부
4 Linux 서버 AD 계정 SSH 로그인 ssh testuser000001@test.local@서버 정상 로그인
5 계정 삭제 → 즉시 차단 확인 AD에서 계정 삭제 후 파일 공유/SSH 재시도 즉시 접근 거부
6 패스워드 정책 적용 짧은 패스워드 변경 시도 정책 위반 오류 반환
7 LDAP 쿼리 (대량 계정 환경) ldapsearch로 10만 계정 조회 응답 시간 측정 3초 이내 응답
8 Kerberos 티켓 발급 및 갱신 kinit / klist 명령으로 티켓 상태 확인 티켓 정상 발급
📋
관리 UI 추가 개발 시 참고 기본 CLI 운영은 samba-tool로 시작하고, 필요 기능이 명확해지면 python-ldap3 기반 REST API + React 포털을 단계적으로 개발하는 것을 권장합니다. 초기부터 풀 UI를 개발하면 불필요한 기능을 만드는 낭비가 생깁니다.

7DNS 설정

Active Directory는 DNS에 강하게 의존합니다. 도메인 조인, Kerberos 인증, LDAP 서비스 탐색 모두 DNS SRV 레코드를 통해 동작하므로 DNS 설정이 잘못되면 AD 전체가 동작하지 않습니다.

ℹ️
AD가 DNS에 의존하는 이유 Windows PC나 Linux 서버가 도메인에 조인할 때 가장 먼저 _ldap._tcp.TEST.LOCAL SRV 레코드를 조회합니다. 이 레코드가 없거나 응답하지 않으면 도메인 조인 자체가 불가능합니다.

자동 생성되는 SRV 레코드

⚠️
자주 발생하는 DNS 문제
  • systemd-resolved 충돌 — 포트 53 선점으로 Samba DNS가 시작 안 됨. systemctl disable --now systemd-resolved 필수
  • Windows PC DNS 설정 미변경 — 도메인 조인 시 "도메인을 찾을 수 없음" 오류의 90%가 이 원인
  • 역방향 조회 존(PTR) 미생성 — Kerberos 인증 실패의 원인이 될 수 있음. samba-tool dns 로 수동 추가 가능

조직 구조 초기화 스크립트

프로비저닝 후 실제 사용할 OU/그룹 구조를 자동으로 세팅하는 스크립트입니다.

init_ad_structure.py 
#!/usr/bin/env python3
"""
Samba 4 AD 조직 구조 초기화 스크립트
프로비저닝 직후 1회 실행 — OU, 그룹, 기본 정책 세팅
사용법: python3 init_ad_structure.py
"""
import subprocess, sys

DOMAIN   = "TEST"
REALM    = "TEST.LOCAL"
BASE_DN  = "DC=TEST,DC=LOCAL"
ADMIN_PW = "Admin@Test1234!"

def run(cmd, check=True):
    r = subprocess.run(cmd, shell=True, capture_output=True, text=True)
    if check and r.returncode != 0:
        print(f"  [WARN] {r.stderr.strip()}")
    return r.returncode == 0

def samba(subcmd):
    return run(f'samba-tool {subcmd} -U Administrator --password="{ADMIN_PW}"')

# ── 1. 조직 OU 생성 ──────────────────────────────────────────
OUS = [
    ("개발팀",     "OU=개발팀"),
    ("마케팅팀",   "OU=마케팅팀"),
    ("인사팀",     "OU=인사팀"),
    ("영업팀",     "OU=영업팀"),
    ("IT인프라팀", "OU=IT인프라팀"),
    ("서버",       "OU=서버"),       # 서버 컴퓨터 계정용
]
print("[1/4] OU 생성")
for name, ou_rdn in OUS:
    samba(f'ou create "{ou_rdn},{BASE_DN}"')
    print(f"  OU: {name}")

# ── 2. 부서 그룹 생성 ─────────────────────────────────────────
GROUPS = [
    ("개발팀",         "OU=개발팀"),
    ("마케팅팀",       "OU=마케팅팀"),
    ("인사팀",         "OU=인사팀"),
    ("영업팀",         "OU=영업팀"),
    ("IT인프라팀",     "OU=IT인프라팀"),
    ("서버관리자",     "OU=IT인프라팀"),  # Linux sudo 권한용
    ("파일공유_읽기",  "OU=IT인프라팀"),
    ("파일공유_쓰기",  "OU=IT인프라팀"),
]
print("[2/4] 그룹 생성")
for grp, ou_rdn in GROUPS:
    samba(f'group add "{grp}" --groupou="{ou_rdn},{BASE_DN}"')
    print(f"  그룹: {grp}")

# ── 3. 패스워드 정책 설정 ────────────────────────────────────
print("[3/4] 도메인 패스워드 정책 적용")
samba('domain passwordsettings set --min-pwd-length=8')
samba('domain passwordsettings set --complexity=on')
samba('domain passwordsettings set --max-pwd-age=90')
samba('domain passwordsettings set --account-lockout-threshold=5')
samba('domain passwordsettings set --account-lockout-duration=30')
print("  완료")

# ── 4. 검증 ─────────────────────────────────────────────────
print("[4/4] 검증")
run(f'samba-tool ou list -U Administrator --password="{ADMIN_PW}"')
run(f'samba-tool group list -U Administrator --password="{ADMIN_PW}"')
print("\n초기화 완료.")
ℹ️
실행 순서 Phase 1 (Samba 프로비저닝) 완료 후 바로 이 스크립트를 실행합니다. 이후 Phase 5 대량 계정 생성 스크립트에서 --userou 옵션으로 각 계정을 적절한 OU에 배치할 수 있습니다.
